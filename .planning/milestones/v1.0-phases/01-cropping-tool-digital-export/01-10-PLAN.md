       - Run `bun vite` dev server, load a photo in the browser
       - Open browser console: `window.__PASSPHOTO_HMR__` — does it contain data?
       - If empty/null: `persistForHmr()` is never called or `import.meta.hot` is falsy
       - If populated: the persist side works, check the restore side

    2. **Check Svelte 5 module re-evaluation behavior:**
       - Svelte 5 with runes may handle `.svelte.ts` files differently from plain `.ts`
       - `$state()` initialization with `_hmr?.crop` may not work if the module isn't truly re-evaluated (Svelte may recreate state differently)
       - Check if `import.meta.hot.accept()` is needed for self-accepting HMR

    3. **Check blob URL survival:**
       - After HMR, the blob URL may be revoked if the previous module instance cleaned up
       - `URL.createObjectURL` blob URLs persist as long as the document is alive, so this is unlikely but verify

    4. **Check `export let` with `$state` and HMR:**
       - Svelte 5's `$state` rune creates reactive proxies. On HMR, the module re-evaluates, creating NEW `$state` instances
       - But Svelte components may still hold references to the OLD proxies
       - This is the most likely cause: components import `cropState` and `imageState` by reference, but HMR creates new objects that components don't see

    **If root cause is Svelte 5 runes + HMR proxy mismatch:**
    Use Vite's `import.meta.hot.accept()` API to manually handle module updates. Instead of relying on module-level re-evaluation:
    ```typescript
    if (import.meta.hot) {
      import.meta.hot.accept((newModule) => {
        // Don't let Vite replace the module — keep existing state
      });
      // Persist on dispose (before new module loads)
      import.meta.hot.dispose(() => {
        persistForHmr();
      });
    }
    ```

    **If root cause is `persistForHmr()` never being called:**
    The function is exported and called from `CropEditor.svelte` after pan/zoom. But it's NOT called after `loadImage()`. Check if `loadImage()` calls `persistForHmr()` — if not, the initial photo load is never persisted.

    **If root cause is something else:**

    - State is persisted to `window.__PASSPHOTO_HMR__` on every mutation (pan, zoom, image load)

    - The Image element is properly reconstructed from the persisted blob URL
